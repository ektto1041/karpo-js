    LLM 기반의 코드 리뷰 주제 제안 서버 구현

    프로젝트: StockHolmes

    목표: WMS 구현

    팀 구성: 프론트엔드 2명, 백엔드 4명

    역할: 팀장 및 프론트엔드 담당

    문제 상황

    팀 내 코드 리뷰 문화가 활성화되지 않아 팀원 간 갈등이 발생했습니다. 초기에는 정기적인 코드 리뷰를 약속하고 프로젝트를 시작했지만, 리뷰 시 실질적인 논의가 거의 없었고, 팀원 간 피드백에 대한 기대도 낮은 상태였습니다.

    팀원들과 개별 대화를 나눈 결과, “내가 틀릴까 봐 말을 아끼게 된다.”, “리뷰해도 별 도움이 안 되는 것 같다.” 와 같은 반응이 있었습니다.

    해결 과정

    이 문제를 단순한 커뮤니케이션 개선이 아닌 시스템적 접근으로 해결하고자 했습니다.

    코드 리뷰는 사람의 지식에 의존하기 때문에 한계가 있습니다. 이에 저는 자동화된 코드 리뷰 주제 제안 서버를 기획하고 직접 구현했습니다.

    먼저 시퀀스 다이어그램을 통해 코드 리뷰 주제 제안 서버의 기능을 기획했습니다.

    간단한 서버를 구현하기 위해 Javascript와 Express.js를 활용하여 개발을 진행했고, open AI API를 활용해서 코드 리뷰 주제를 생성하도록 구현했습니다. 이후 AWS EC2를 활용해 서버를 배포했고, Github Webhook과 연동했습니다.

    결과

    코드 리뷰 주제 제안 서버의 도입은 팀 내 코드 리뷰 문화에 실질적인 변화를 가져왔습니다. 팀원들이 제안받은 주제로 각자 의견을 내기 시작했고, 이를 통해 여러가지 개선점을 알아낼 수 있었습니다.

    먼저 프론트엔드에서는 복잡했던 Table 컴포넌트를 리팩토링 할 수 있었습니다. Table을 그리기 위한 기본 로직을 다루는 useTable 훅, 각 데이터의 특성에 맞게 가공하는 useDataTable 훅 그리고 실제 Table의 UI를 가지고 있는 Table 컴포넌트로 분리했고, 이를 통해 중복 코드 제거 및 유지보수 용이성 향상 등의 개선을 이뤄냈습니다.

    백엔드에서는 MyBatis를 활용해 데이터를 INSERT하는 과정에서 대량의 INSERT문을 batch insert로 수정하여 성능을 개선했습니다.

    회고

    이번 경험을 통해 기술적 해결책을 통해 팀 문화를 바꿀 수 있음을 느꼈습니다. 단순히 기능 구현을 넘어서, 시스템적으로 팀의 생산성과 협업 문화를 개선하는 방법에 대해 고민하고 적용할 수 있었던 의미 있는 경험이었습니다.

    SSE로 구현한 실시간 알림

    프로젝트: Gasannes

    목표: 영화 정보/리뷰 사이트 구현

    팀 구성: 프론트엔드 2명, 백엔드 3명

    역할: 팀장 및 프론트엔드 담당

    문제 상황

    Gasannes는 사용자가 영화에 투표하고, 주기적으로 열리는 시상식 결과를 실시간으로 확인할 수 있는 기능을 중심으로 설계된 서비스입니다. 영화 리뷰와 평점 제공이라는 기본 기능 위에 '유저 참여형 시상식'이라는 차별화된 콘셉트를 더해 사용자 몰입도를 높이는 것이 핵심 목표였습니다.

    해결 과정

    우선 시상식 이벤트를 자동으로 발생시키기 위해 Spring Scheduler를 도입했습니다. 이를 통해 서버가 매주 특정 시간에 시상식을 시작할 수 있도록 설정했습니다.

    다음으로, 실시간 알림 기능을 구현하기 위해 세 가지 기술을 검토했습니다.

    첫 번째 후보였던 Polling은 클라이언트가 일정 주기로 서버에 요청을 보내는 방식이지만, 서버에 불필요한 부담이 크기 때문에 적합하지 않다고 판단했습니다.

    두 번째는 WebSocket이었는데, 이는 양방향 통신이 가능하다는 장점이 있지만, 이 프로젝트에서는 클라이언트에서 서버로 메시지를 보낼 필요가 없었기 때문에 채택하지 않았습니다.

    마지막으로 Server-Sent Events(SSE)를 고려했습니다. SSE는 서버가 클라이언트에 실시간으로 단방향 메시지를 보낼 수 있는 기술로, 이 프로젝트의 요구사항과 가장 잘 맞아 최종적으로 SSE를 채택하여 구현했습니다.

    결과

    Spring Scheduler를 통해 주기적으로 시상식 이벤트가 발생하도록 설정했고, 서버는 SSE 채널을 통해 클라이언트에 실시간으로 이벤트 발생 사실을 알릴 수 있게 되었습니다. 사용자는 알림을 받고 시상식 결과를 즉시 확인할 수 있게 되었으며, 시상식에 대한 참여도와 흥미를 높이는 데 효과적이었습니다.

    회고

    이번 프로젝트에서는 서버에서 먼저 클라이언트에 데이터를 전달하는 구조를 처음으로 구현해보며, 기존에 경험하지 못한 새로운 통신 방식의 흐름을 익힐 수 있었습니다. 단방향 푸시 기반 구조는 알림이나 실시간 기능에 매우 적합하다는 점을 실감했고, 기술을 선택할 때 단순 구현 편의성뿐만 아니라 실제 서비스 요구사항에 얼마나 적합한지를 함께 고려해야 한다는 점을 배웠습니다.

    UX 중심 프론트엔드 개발 경험

    프로젝트: StockHolmes

    목표: WMS 구현

    팀 구성: 프론트엔드 2명, 백엔드 4명

    역할: 팀장 및 프론트엔드 담당

    문제 상황

    StockHolmes는 대시보드와 대용량 표 등 정보를 시각적으로 효과적으로 보여주는 기능이 핵심인 프로젝트였습니다. 사용자 경험을 높이기 위한 설계와 구현이 중요한 과제라고 생각했습니다.

    특히 대시보드와 테이블 컴포넌트는 데이터의 양이 많고 시각적 요소가 복잡해, 로딩 시간 지연이나 예외 발생 시 사용자에게 충분한 피드백을 제공하지 않으면 UX가 크게 저하될 수 있는 구조였습니다. 이를 해결하기 위해 사용자의 흐름을 방해하지 않으면서도 명확한 상태 정보를 제공하는 방법을 고민했습니다.

    해결 과정 1: 사용자에게 정확한 피드백 제공하기

    우선 API 요청에 실패하거나 로딩이 지연될 경우, 사용자에게 명확한 상태 전달이 필요하다고 판단했습니다. 이에 따라 UI 설계 단계에서부터 Error Boundary의 적용 범위를 구체적으로 계획했고, 모든 주요 컴포넌트에 예외 처리를 포함시켰습니다.

    예외 발생 시 Error Boundary의 fallback UI를 통해 오류 메시지와 함께 새로고침 버튼을 제공해, 사용자가 즉시 대응할 수 있도록 했습니다.

    또한 Suspense를 도입하여 데이터 로딩 중에는 스켈레톤 UI나 스피너를 통해 사용자가 현재 어떤 상태에 있는지 명확하게 인지할 수 있도록 했습니다.

    해결 과정 2: 번들 사이즈 최적화를 통한 초기 로딩 개선

    대시보드에 사용된 차트는 대부분 Nivo 라이브러리를 기반으로 구현되었습니다. 그러나 Nivo는 기능이 풍부한 대신 번들 사이즈가 커서 초기 로딩에 부담을 줄 수 있다는 점이 문제였습니다.

    이를 개선하기 위해 Nivo 컴포넌트를 Lazy Import 방식으로 전환했습니다. 필요한 시점에만 동적으로 로드되도록 하여 초기 렌더링 시 불필요한 자원 소비를 줄였고, 실제로 빌드 후 번들 용량이 절반 수준으로 줄어드는 효과를 얻을 수 있었습니다.

    결과

    ErrorBoundary와 Suspense를 통해 예외 상황과 로딩 상태에 대한 사용자 피드백을 개선했고, "로딩 여부를 직관적으로 알 수 있어서 좋다"는 피드백을 받았습니다.

    또한 Lazy Import 도입 전 1,000kb에 달하던 번들 사이즈를 약 500kb까지 줄이며 성능 최적화 성과를 달성했습니다.

    회고

    이번 프로젝트를 통해 웹 애플리케이션 개발에서 사용자 경험을 고려한 설계가 얼마나 중요한지를 체감했습니다. 단순히 기능을 구현하는 것에서 그치지 않고, 사용자 입장을 고려하는 개발이 서비스의 완성도를 높인다는 사실을 배웠습니다.

    아쉬운 점은 프로젝트의 규모가 비교적 작아, 번들 사이즈 최적화에 따른 체감 성능 차이가 다소 제한적이었다는 것입니다. 다음에는 더 큰 규모의 애플리케이션에서 이러한 최적화 효과를 실제로 체감해볼 수 있는 경험을 쌓고 싶다는 생각을 하게 되었습니다.
